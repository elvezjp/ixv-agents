# 比較レポート：仕様・コード整合性レビュースキル vs 外部ツール

このレポートでは、本スキル（`spec-code-reviewer-skill`）と外部ツール（`elvezjp/spec-code-ai-reviewer`）を比較し、その能力と違いを明らかにします。

## 1. 比較の概要

外部のGitHubツールは、日本の現場で多い**Excel設計書のバッチ処理に特化したWebアプリケーション**です。対して、本**スキル**は、開発フローの中で対話的かつ深層的なレビューを提供し、そのまま修正まで実行できる**AIエージェント（パートナー）**です。

| 特徴 | `elvezjp/spec-code-ai-reviewer` | 本スキル (`spec-code-reviewer-skill`) |
| :--- | :--- | :--- |
| **コンセプト** | バッチ処理ツール (Webアプリ) | 対話型AIパートナー (エージェント) |
| **主な入力ソース** | **Excel (.xlsx)** | **Markdown / ソースコード / リポジトリ全体** |
| **レビュー手法** | 固定されたプロンプトによる一括比較 | 状況に応じた探索と論理的推論 |
| **アクション** | レポート生成のみ | レポート生成 ＋ **コードの直接修正** |
| **文脈の理解** | アップロードされたファイルに限定 | リポジトリ全体（設定、ライブラリ等）を把握 |

## 2. 本スキルの主な強み

### 1) 「なぜ（Why）」を追求する推論能力
GitHubツールが「差分」を見つけることに集中するのに対し、本スキルは**実装の意図**を分析します。コードが仕様と異なる場合、単に指摘するだけでなく「なぜそうなったのか」を推測し、仕様自体の曖昧さを指摘したり、ユーザーに逆質問したりすることで、より本質的な問題を解決します。

### 2) 自律的なリポジトリ探索
本スキルはリポジトリ内を自律的に移動できます。例えば、仕様書が別のファイルで定義されたデータモデルを参照している場合、エージェントは自らそのファイルを読みに行き、一貫性を確認します。一方、バッチツールではあらかじめ全ての文脈を読み込ませる必要があります。

### 3) ユーザーとの対話型ワークフロー
本スキルは人間と協力して動きます。静的なレポートを出して終わりではなく、人間のフィードバックを受けて内容を精査し、納得感のあるレビューを作り上げます。

### 4) 即時の修正実行
レポートを出すだけでなく、発見した問題に対してツールを使い、**その場で修正案を作成し、実際にファイルを更新する**までを一気通貫で行えます。

## 3. 限界と対策

*   **Excelの直接パース能力**: GitHubツールは、結合セルなど複雑なExcel構造を読み解く専用エンジン（`excel2md`）を持っています。
    *   *対策*: 最高精度のレビューを行うため、設計書がExcelの場合はMarkdownやCSV形式に書き出したものを渡していただくことを推奨します。
*   **大量のファイル一括処理**: 数百ファイルを同時に処理する場合は、専用サーバーを持つWebアプリ形式が有利です。
    *   *対策*: 本スキルは「機能単位」や「モジュール単位」での質の高いレビュー、および「修正まで完結させる」ワークフローに最適化されています。

## 4. 結論

本スキルは、AIエージェントの対話的特性を活かすことで、対象ツールと**同等、あるいはそれ以上の深さのレビュー**を提供します。「問題を見つける」だけでなく、「コンテキストを理解し、共に解決する」パートナーを求める開発者にとって、非常に強力なツールとなります。
