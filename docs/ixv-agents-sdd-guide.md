## 1. 仕様駆動開発とは

IXV-Agents は「仕様（Spec）」を唯一の信頼できる情報源（Single Source of Truth）として開発を進めるシステムです。

### 4つの原則

1. **仕様はリビングドキュメントである** — 仕様は固定ではなく、開発の進行に合わせて更新される
2. **仕様は唯一の信頼できる情報源（SSoT）である** — すべての判断は仕様（`workspace/README.md`）に基づく
3. **変更と反復を前提とする** — 計画は段階的に実行され、フィードバックで改善される
4. **AIがコストを削減し、人間が判断する** — AIエージェントが作業し、最終判断は人間が行う

### ユーザーの役割

ユーザー（人間）の役割は以下の3つです：

- **要望を伝える** — PO エージェントに作りたいものを伝える
- **承認する** — 仕様、計画、成果物を確認して承認する
- **判断する** — 問題が発生した場合に方針を決める

コードを書いたり、タスクを管理したりする必要はありません。すべてエージェントチームが行います。

## 2. エージェントチームの構成

IXV-Agents は 5 体のエージェントで構成される固定チームです。

### Product Owner (PO) — 1体

| 項目 | 内容 |
|------|------|
| 責任 | プロダクトの価値最大化 |
| 主な業務 | ユーザーとの対話、仕様書の作成・更新指示、受入判断 |
| 書き込み可能 | `queue/po_to_sm.yaml`, `workspace/README.md` |

**ユーザーが直接会話する唯一のエージェントです。** ユーザーの要望を受け取り、SM に指示を出します。実装やタスク管理は行いません。

### Scrum Master (SM) — 1体

| 項目 | 内容 |
|------|------|
| 責任 | プロセス管理と障害除去 |
| 主な業務 | タスク分解、Dev への割り当て、ダッシュボード管理、仕様書の更新 |
| 書き込み可能 | `queue/tasks/*.yaml`, `queue/dashboard.md`, `workspace/README.md`, `CONSTITUTION.md`, `docs/*` |

PO からの指示を受けてタスクに分解し、Dev に振り分けます。プロジェクト全体の進捗を `queue/dashboard.md` で管理します。

### Development (Dev) — 3体

| 項目 | 内容 |
|------|------|
| 責任 | 動作するソフトウェアの作成 |
| 主な業務 | 設計・コーディング・テスト・調査 |
| 書き込み可能 | `queue/reports/*.yaml`, 実装関連ファイル |

SM から割り当てられたタスクを実装します。各 Dev は専用のタスクファイル（`queue/tasks/dev1.yaml` 等）を持ち、完了後に報告ファイル（`queue/reports/*.yaml`）を作成します。

### 指揮系統

```
ユーザー（人間）
  │ 要望・承認
  ▼
  PO（プロダクトオーナー）
  │ 指示 (queue/po_to_sm.yaml)
  ▼
  SM（スクラムマスター）
  │ タスク割り当て (queue/tasks/dev{N}.yaml)
  ▼
  Dev1 / Dev2 / Dev3（開発者）
  │ 報告 (queue/reports/{task_id}.yaml)
  ▼
  SM → PO → ユーザー
```

- ユーザーは PO とのみ会話する
- PO は SM にのみ指示を出す（Dev に直接指示しない）
- Dev は SM にのみ報告する（PO やユーザーに直接連絡しない）

## 3. システムの仕組み

### 3.1. tmux セッションとペイン配置

エージェントは 1 つの tmux セッション（`ixv-agents`）上で動作します。起動すると、以下のレイアウトで 5 つのペインが配置されます。

```
[ixv-agents] All Agents (5 panes)
┌───────────────┬───────┬───────┬───────┐
│               │       │       │       │
│      PO       │ Dev1  │ Dev2  │ Dev3  │
│    (0.0)      │ (0.2) │ (0.3) │ (0.4) │
├───────────────┤       │       │       │
│               │       │       │       │
│      SM       │       │       │       │
│    (0.1)      │       │       │       │
└───────────────┴───────┴───────┴───────┘
  左50%            右50%を3等分
```

| ペイン | 役割 | ユーザーの操作 |
|--------|------|---------------|
| 0.0 (左上) | PO | **ここに要望を入力する** |
| 0.1 (左下) | SM | 自動動作（操作不要） |
| 0.2 | Dev1 | 自動動作（操作不要） |
| 0.3 | Dev2 | 自動動作（操作不要） |
| 0.4 | Dev3 | 自動動作（操作不要） |

**ユーザーが操作するのは PO ペイン（左上）のみです。** SM や Dev のペインは進捗の確認のために閲覧できますが、入力する必要はありません。

### 3.2. ワークスペース

`workspace/` ディレクトリは、エージェントが実際に作業を行う隔離された環境です。リポジトリルートとは分離されており、ツール自体の設定ファイルにエージェントがアクセスすることを防ぎます。

```
workspace/
├── README.md           # プロジェクト仕様（SSoT）
├── CONSTITUTION.md     # プロジェクト憲章
├── PROCESS.md          # 7つの工程と運用フロー
├── AGENTS.md           # AI行動規範
├── roles/              # ロール指示書（symlink → ../roles）
├── queue/              # エージェント間通信
│   ├── dashboard.md    # プロジェクトステータスボード
│   ├── po_to_sm.yaml   # PO → SM への指示
│   ├── tasks/          # SM → Dev へのタスク
│   │   ├── dev1.yaml
│   │   ├── dev2.yaml
│   │   └── dev3.yaml
│   └── reports/        # Dev → SM への報告
│       └── TASK-*.yaml
└── (成果物)             # 実装コード、テスト、ドキュメント等
```

#### 主要ファイル

| ファイル | 役割 | 管理者 |
|---------|------|--------|
| `README.md` | プロジェクト仕様（唯一の信頼できる情報源） | PO が指示、SM が更新 |
| `CONSTITUTION.md` | プロジェクト憲章（存在意義と基本原則） | SM が更新、人間が承認 |
| `queue/dashboard.md` | プロジェクト全体の進捗状況 | SM が更新 |
| `queue/po_to_sm.yaml` | PO から SM への指示 | PO が書き込み |
| `queue/tasks/dev{N}.yaml` | SM から各 Dev へのタスク | SM が書き込み |
| `queue/reports/*.yaml` | Dev から SM への報告 | Dev が書き込み |

### 3.3. エージェント間通信（queue/）

エージェント間の通信は、YAML ファイルと tmux の `send-keys` によるイベント駆動で行われます。

```
PO                          SM                          Dev
 │                           │                           │
 │  queue/po_to_sm.yaml      │                           │
 │ ─────────────────────►    │                           │
 │                           │  queue/tasks/dev{N}.yaml  │
 │                           │ ─────────────────────►    │
 │                           │                           │
 │                           │  queue/reports/{id}.yaml  │
 │                           │ ◄─────────────────────    │
 │                           │                           │
 │  send-keys（通知）         │  send-keys（通知）         │
 │ ◄─────────────────────    │ ◄─────────────────────    │
```

1. **PO → SM**: `queue/po_to_sm.yaml` に指示を書き込み、`send-keys` で SM を起動
2. **SM → Dev**: `queue/tasks/dev{N}.yaml` にタスクを書き込み、`send-keys` で Dev を起動
3. **Dev → SM**: `queue/reports/{task_id}.yaml` に報告を書き込み、`send-keys` で SM を起動
4. **SM → PO**: `queue/dashboard.md` を更新し、`send-keys` で PO に完了通知

エージェントはポーリング（待機ループ）をせず、通知を受けて動き出します。これにより API コストを最小限に抑えます。

#### トレーサビリティ

すべての指示・タスク・報告は ID で追跡できます。

| ID形式 | 用途 | 例 |
|--------|------|-----|
| `REQ-YYYYMMDD-###` | PO からの要求 | `REQ-20260201-001` |
| `TASK-YYYYMMDD-###` | SM が分解したタスク | `TASK-20260201-004` |
| `spec_ref` | 仕様書への参照 | `README.md#requirements` |

## 4. 開発の流れ — 7つの工程

開発は 7 つの工程を順に進みます。各工程には明確な入力・出力・承認者が定義されています。

```
工程1: 原則決定 ──► 工程2: 企画・要件定義 ──► 工程3: 設計計画
                                                    │
                                                    ▼
工程7: 移行・運用 ◄── 工程6: 検証・受入 ◄── 工程5: 実装 ◄── 工程4: タスク分割
       │
       ├─ 仕様更新が必要 → 工程2 へ
       └─ 仕様変更不要   → 工程4 へ
```

### 工程1. 原則決定（Constitution）

プロジェクトの存在意義と基本原則を定める工程です。**初回のみ実行されます。**

#### ユーザーの操作

1. PO ペインに **プロジェクトの目的** を伝える
2. PO が SM に憲章の更新を指示する
3. SM が `CONSTITUTION.md` を更新する
4. PO がユーザーに承認を依頼する
5. **ユーザーが `CONSTITUTION.md` を確認して承認する**

#### 流れ

```
[ユーザー] プロジェクトの目的を PO に伝える
    ↓
[PO] CONSTITUTION.md を確認
    ├─ 目的が記入済み → スキップ（工程2へ）
    └─ 目的が未記入
         ↓
     [PO] SM に憲章更新を依頼（po_to_sm.yaml）
         ↓
     [SM] CONSTITUTION.md を更新
         ↓
     [PO] ユーザーに承認を依頼
         ↓
     [ユーザー] ★ 憲章を確認して承認
```

#### この工程の成果物

| ファイル | 担当 |
|---------|------|
| `CONSTITUTION.md` | SM が更新、**ユーザーが承認** |
| `queue/dashboard.md`（Current Phase 更新） | SM |

---

### 工程2. 企画・要件定義（Specify）

ユーザーの要望を仕様書（`README.md`）に反映する工程です。

#### ユーザーの操作

1. PO ペインに **作りたいもの・実現したいこと** を伝える
2. PO が要望を整理し、SM に仕様策定を指示する
3. SM が `README.md`（仕様書）を更新する
4. PO がユーザーに承認を依頼する
5. **ユーザーが仕様を確認して承認する**

#### 流れ

```
[ユーザー] 要望を PO に伝える
    ↓
[PO] README.md を確認
    ├─ 要望が反映済み → スキップ（工程3へ）
    └─ 要望が未反映
         ↓
     [PO] SM に仕様策定を依頼（po_to_sm.yaml）
         ↓
     [SM] README.md を更新（要件、受入基準、バックログ等）
         ↓
     [PO] ユーザーに承認を依頼
         ↓
     [ユーザー] ★ 仕様を確認して承認
```

#### この工程の成果物

| ファイル | 担当 |
|---------|------|
| `README.md`（仕様書） | SM が更新、**ユーザーが承認** |
| `queue/dashboard.md`（Current Phase 更新） | SM |

#### ポイント

- `README.md` がこのプロジェクトにおける**唯一の仕様書**です
- 要件、受入基準、制約、バックログがここに集約されます
- 仕様に記載されていない内容は仕様として扱われません

---

### 工程3. 設計計画（Plan）

SM が実行計画を策定する工程です。要件の複雑さに応じて、SM が直接計画するか、Dev に調査を依頼します。

#### ユーザーの操作

1. PO が SM に計画策定を依頼する（前工程から自動的に進む場合もある）
2. SM が計画を策定する（複雑な要件の場合は Dev に調査を依頼）
3. SM が `docs/` に計画書を作成し、`README.md` を詳細化する
4. PO がユーザーに計画の承認を依頼する
5. **ユーザーが計画を確認して承認する**

#### 流れ

```
[PO] SM に計画策定を依頼（po_to_sm.yaml）
    ↓
[SM] 要件の複雑さを判断
    ├─ 単純な要件 → SM が直接計画を策定
    └─ 複雑な要件
         ↓
     [SM] Dev に調査・検討タスクを割り当て
         ↓
     [Dev] 調査を実施し結果を報告
         ↓
     [SM] 調査結果を踏まえて計画を策定
    ↓
[SM] docs/ に計画書を作成、README.md を詳細化
    ↓
[PO] ユーザーに承認を依頼
    ↓
[ユーザー] ★ 計画を確認して承認
```

#### この工程の成果物

| ファイル | 担当 |
|---------|------|
| `docs/*`（計画書・設計メモ） | SM が作成、**ユーザーが承認** |
| `README.md`（仕様の詳細化） | SM |
| `queue/dashboard.md`（Current Phase 更新） | SM |

#### ポイント

- 計画書は `docs/` に作成されますが、あくまで補助資料です
- **仕様（SSoT）は `README.md` のみ**です。重要な決定は `README.md` に反映されます
- 計画が大規模な場合、段階的な実行計画（フェーズ分け）になることがあります

---

### 工程4. タスク分割（Tasks）

SM がタスクを分解し、Dev に割り当てる工程です。

#### ユーザーの操作

- **この工程ではユーザーの操作は不要です。** PO が計画に基づいて実行を指示し、SM がタスクを分割します。

#### 流れ

```
[PO] SM に実行を指示（po_to_sm.yaml）
    ↓
[SM] 計画書（docs/）を読み取り
    ↓
[SM] 今回実施する範囲を判断
    ↓
[SM] タスクを分解し queue/tasks/dev{N}.yaml を作成
    ↓
（工程5へ）
```

#### この工程の成果物

| ファイル | 担当 |
|---------|------|
| `queue/tasks/dev{N}.yaml` | SM |
| `queue/dashboard.md`（Backlog Status 更新） | SM |

#### ポイント

- SM は独立したタスクは可能な限り複数の Dev に並列で割り当てます
- 依存関係のあるタスクは順次実行されます
- 同一ファイルへの同時書き込みを避けるため、各 Dev には別々のファイルが割り当てられます

---

### 工程5. 実装（Implement）

Dev がタスクに従い実装を行う工程です。

#### ユーザーの操作

1. SM が Dev にタスクを通知し、Dev が実装を開始する（自動）
2. Dev が完了すると SM に報告し、SM が PO に報告する（自動）
3. PO がユーザーに成果物を報告する
4. **ユーザーが成果物を確認し、次のアクションを指示する**

#### 流れ

```
[SM] Dev にタスクを通知（send-keys）
    ↓
[Dev] タスクを読み取り、実装作業を実施
    ↓
[Dev] 報告書を作成（queue/reports/{task_id}.yaml）
    ↓
[Dev] SM に完了を通知（send-keys）
    ↓
[SM] 報告を確認し、dashboard.md を更新
    ↓
[SM] PO に完了内容を報告（send-keys）
    ↓
[PO] ユーザーに報告
    ↓
[ユーザー] 成果物を確認
    ├─ 問題なし、計画完了 → 工程6（検証・受入）へ
    ├─ 問題なし、計画未完了 → 工程4（タスク分割）へ（次フェーズ）
    └─ 問題あり → 修正を指示 → 工程4（タスク分割）へ
```

#### この工程の成果物

| ファイル | 担当 |
|---------|------|
| 実装コード、テスト等 | Dev |
| `queue/reports/{task_id}.yaml` | Dev |
| `queue/dashboard.md`（Agent Status 更新） | SM |

#### ポイント

- 段階的な計画の場合、ユーザーは各フェーズの完了後に次に進むかを判断します
- 問題が発生した場合は PO がユーザーに報告し、修正方針の判断を求めます
- Dev がブロッカーに遭遇した場合は `status: blocked` で報告され、SM が解決策を検討します

---

### 工程6. 検証・受入（Verify/Accept）

成果物を仕様と照合し、受け入れるかどうかを判断する工程です。

#### ユーザーの操作

1. PO が SM に検証開始を依頼する
2. SM が仕様と計画に基づいて成果物を検証する
3. PO が SM の検証結果と仕様に基づいて判断する
4. **問題なし → ユーザーに最終承認を依頼する**
5. **問題あり → ユーザーに問題を報告し、工程3に差し戻す**

#### 流れ

```
[PO] SM に検証開始を依頼（po_to_sm.yaml）
    ↓
[SM] 計画（docs/）と仕様（README.md）に基づき成果物を検証
    ↓
[SM] PO に検証結果を報告（send-keys）
    ↓
[PO] SM の報告と仕様に基づき検証・判断
    ├─ 問題なし
    │      ↓
    │  [PO] ユーザーに受入の承認を依頼
    │      ↓
    │  [ユーザー] ★ 最終承認
    │      ↓
    │  [SM] README.md の Backlog ステータスを done に更新
    │
    └─ 問題あり
           ↓
       [PO] ユーザーに問題を報告
           ↓
       [ユーザー] ★ 工程3（設計計画）へ差し戻し
```

#### この工程の成果物

| ファイル | 担当 |
|---------|------|
| `README.md`（Backlog ステータス更新） | SM |
| `queue/dashboard.md`（Current Phase 更新） | SM |

### 工程7. 移行・運用（Migration/Operation）

運用中のフィードバックを受けて、次のアクションを決める工程です。

#### ユーザーの操作

1. **運用中に気付いたフィードバック（バグ、改善要望等）を PO に伝える**
2. SM がフィードバックを分析し、適切な工程へ振り分ける

#### 流れ

```
[ユーザー] フィードバックを PO に伝える
    ↓
[PO] SM にフィードバックを伝達（po_to_sm.yaml）
    ↓
[SM] フィードバックを分析
    ├─ 仕様更新が必要 → 工程2（企画・要件定義）へ
    └─ 仕様変更不要（バグ修正、調査等） → 工程4（タスク分割）へ
```

## 5. ユーザー操作ガイド

### 5.1. PO への指示の出し方

エージェント起動後、PO ペイン（左上）に自然言語で要望を入力します。

#### 初回起動時の例

```
「ToDoアプリを作りたい。タスクの追加・削除・完了ができて、
ローカルストレージに保存されるようにしてほしい。」
```

PO はこの要望を受けて、以下の工程を自動的に進めます：

1. `CONSTITUTION.md` を確認 → 未記入なら憲章策定（工程1）
2. `README.md` を確認 → 要望が未反映なら仕様策定（工程2）
3. 計画策定（工程3）→ タスク分割（工程4）→ 実装（工程5）

#### 追加の要望・修正指示の例

```
「完了したタスクを一括削除する機能を追加してほしい。」
```

```
「UIのデザインを変更したい。ダークモードに対応してほしい。」
```

```
「バグがある。タスクを追加した後にページを更新すると消える。」
```

### 5.2. 進捗の確認方法

#### dashboard.md を確認する

プロジェクトの進捗状況は `workspace/queue/dashboard.md` に集約されます。PO ペインでの会話で状況を聞くこともできますが、直接ファイルを確認することもできます。

dashboard.md には以下の情報が含まれます：

- **Current Phase** — 現在の工程
- **Backlog Status** — 要求の進捗状況
- **Agent Status** — 各エージェントの稼働状況
- **Blockers** — 作業を阻害している問題
- **要対応** — ユーザーの判断が必要な事項

#### 各ペインの確認

tmux のペインを切り替えることで、各エージェントの動作状況を直接確認できます。ただし、SM や Dev のペインへの入力は不要です。

### 5.3. 承認の流れ

開発プロセスの中で、ユーザーの承認が必要なタイミングがあります。

| 工程 | 承認対象 | 確認すべきファイル |
|------|---------|------------------|
| 工程1 原則決定 | プロジェクト憲章 | `CONSTITUTION.md` |
| 工程2 企画・要件定義 | 仕様 | `README.md` |
| 工程3 設計計画 | 計画内容 | `docs/*` |
| 工程5 実装 | 成果物 | 実装コード・テスト等 |
| 工程6 検証・受入 | 検証結果 | `README.md`（Backlog） |

PO が承認を依頼してきたら、対象のファイルを確認し、PO ペインで承認の旨を伝えます。

#### 承認の例

```
「仕様を確認しました。問題ありません。承認します。」
```

#### 差し戻しの例

```
「受入基準にレスポンシブ対応を追加してほしい。修正してから再度確認させてください。」
```

### 5.4. tmux の基本操作

| 操作 | コマンド |
|------|---------|
| ペインの切り替え | `Ctrl+b` → 矢印キー |
| セッションからデタッチ | `Ctrl+b d` |
| セッションに再アタッチ | `tmux attach-session -t ixv-agents` |
| セッション一覧 | `tmux ls` |
| ペインのスクロール | `Ctrl+b [` → 矢印キー/Page Up/Down → `q` で終了 |

デタッチしてもエージェントはバックグラウンドで動作し続けます。再アタッチすればいつでも状況を確認できます。

## 6. ワークスペースのライフサイクル

### 初期化

初回起動時（`boot.sh` / `boot.ps1`）に、ワークスペースが自動的に初期化されます。手動で初期化する場合は以下のコマンドを使用します。

```bash
# macOS
./scripts/setup_workspace.sh

# Windows
.\scripts\setup_workspace.ps1
```

初期化時に `templates/` ディレクトリからテンプレートファイルがコピーされ、シンボリックリンクが作成されます。

### 開発

エージェントは `workspace/` 内で作業を行います。成果物（コード、テスト、ドキュメント等）はすべて `workspace/` 内に作成されます。

### バックアップ

ワークスペースを再初期化する際、既存のデータは自動的に `backups/` にバックアップされます。

```bash
# バックアップ付き再初期化
./scripts/setup_workspace.sh

# バックアップなしで再初期化
./scripts/setup_workspace.sh --no-backup
```

バックアップは `backups/backup_YYYYMMDD_HHMMSS/` の形式で保存されます。

### セッションの停止

```bash
# macOS
./scripts/stop.sh

# プロセスが残る場合は強制停止
./scripts/stop.sh --force
```

## 7. 開発サイクルの全体像

典型的な開発サイクルを、ユーザーの操作を中心にまとめます。

```
1. エージェントを起動する
   $ ./scripts/boot.sh --claude-code

2. PO ペイン（左上）に要望を伝える
   > 「○○を作りたい」

3. [自動] PO が憲章を確認し、必要なら更新を依頼する
   → ★ ユーザーが憲章を承認する

4. [自動] PO が仕様を策定する
   → ★ ユーザーが仕様（README.md）を承認する

5. [自動] SM が計画を策定する
   → ★ ユーザーが計画（docs/*）を承認する

6. [自動] SM がタスクを分割し、Dev に割り当てる

7. [自動] Dev が実装する
   → ★ ユーザーが成果物を確認する
      ├─ 問題なし → 検証へ
      ├─ 次フェーズへ → 6. に戻る
      └─ 修正指示 → 6. に戻る

8. [自動] SM・PO が検証する
   → ★ ユーザーが最終承認する

9. 追加の要望があれば 2. に戻る
```

## 8. Tips

### PO への伝え方のコツ

- **具体的に伝える**: 「いい感じのアプリ」ではなく「タスクの追加・削除・完了ができるToDoアプリ」
- **受入基準を伝える**: 「こういう動作になればOK」という基準を示すと、仕様が的確になる
- **一度に多くを求めすぎない**: 段階的に機能を追加していく方が品質が安定する

### 進捗を確認したいとき

- PO に「現在の進捗は？」と聞く
- `workspace/queue/dashboard.md` を直接確認する
- SM や Dev のペインを目視で確認する

### 問題が発生したとき

エージェントが停止していたり、期待通りに動かない場合は：

1. PO ペインで状況を聞く
2. `workspace/queue/dashboard.md` の Blockers セクションを確認する
3. 必要に応じてセッションを停止して再起動する

### ワークスペースをリセットしたいとき

```bash
./scripts/stop.sh
./scripts/setup_workspace.sh
./scripts/boot.sh --claude-code
```

これにより既存のワークスペースがバックアップされ、新しい状態で開発を始められます。
